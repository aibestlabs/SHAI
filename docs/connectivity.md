# Connectivity Layer

The connectivity layer sits **below** the harness at the network and process boundary — the layer the harness cannot reach. It is a planned future package (`harness-connectivity`). The interface is defined here so the harness is designed to accommodate it from the start.

---

## The bypass problem

The harness gates tool calls before dispatch. Once a call is allowed and dispatched, the actual network traffic is outside the harness's visibility. A tool — or code generated by the LLM and executed by a code-execution tool — can make arbitrary outbound calls: raw HTTP, subprocess exec, direct socket. The harness never sees these.

The connectivity layer catches them by enforcing at the OS/network boundary.

```
agent
  │
  ▼
harness (gates at the API level)
  │
  ▼
connectivity layer (gates at the network level)
  │
  ▼
internet / internal services
```

---

## Dispatch token — the interface

When `check_tool_call` returns `GateDecision(allowed=True)`, the harness issues a signed **dispatch token**. This is the only interface between the harness and the connectivity layer.

```
DispatchToken:
  agent_id:             str
  sub_agent_id:         str | None
  tool_name:            str
  allowed_destinations: list[str]   # URLs / hostnames this call may reach
  issued_at:            datetime
  ttl_seconds:          int         # short — typically 5–30 seconds
  signature:            str         # HMAC-SHA256, shared secret
```

The token is passed to the tool at dispatch time. The connectivity layer's egress proxy validates the token before forwarding the outbound request. Requests without a valid token are dropped.

### Token validation (HMAC-SHA256)

```python
import hashlib, hmac, json, time
from datetime import datetime, timezone

SHARED_SECRET = b"..."   # set in harness.yaml, shared with connectivity layer

def sign_token(payload: dict, secret: bytes) -> str:
    body = json.dumps(payload, sort_keys=True).encode()
    return hmac.new(secret, body, hashlib.sha256).hexdigest()

def verify_token(token: dict, secret: bytes) -> bool:
    sig = token.pop("signature")
    expected = sign_token(token, secret)
    token["signature"] = sig   # restore
    if not hmac.compare_digest(sig, expected):
        return False
    issued = datetime.fromisoformat(token["issued_at"])
    age = (datetime.now(timezone.utc) - issued).total_seconds()
    return age <= token["ttl_seconds"]
```

HMAC-SHA256 is used rather than asymmetric signatures because:
- The verifier (connectivity layer) and signer (harness) run in the same trust boundary.
- Symmetric MAC is faster and simpler to rotate.
- The short TTL limits replay window without needing nonces.

## Egress gateway

The egress gateway is a reverse proxy that sits in front of all outbound connections. It:

1. Intercepts every outbound connection from the agent process.
2. Extracts the dispatch token from the request header or connection metadata.
3. Validates the signature and TTL.
4. Checks `allowed_destinations` against the target host.
5. Forwards on success; drops and logs on failure.

The gateway runs as a sidecar in Kubernetes or as a local transparent proxy for development.

---

## L7 policy

For MCP-connected tools, the connectivity layer can enforce L7 policies — inspecting the content of HTTP requests before forwarding:

- Payload size limits
- Content-type validation
- Response schema enforcement
- Rate limiting per `agent_id`

L7 policies are declared in `harness-connectivity` configuration, not in `harness.yaml`. They are enforced independently of the harness policy rules.

---

## Process isolation

For code-execution tools (e.g. a Python REPL), the connectivity layer enforces process isolation:

- The tool runs in a separate process or container.
- The process has no direct network access — all outbound connections route through the egress gateway.
- The dispatch token scopes which destinations the process may reach.
- The process is destroyed after the tool call completes.

---

## Status in core

The dispatch token interface is not yet implemented in `harness` core. `check_tool_call` currently returns `GateDecision` without a token. When `harness-connectivity` is added:

1. `GateDecision` gains an optional `dispatch_token: str | None = None` field.
2. `check_tool_call` generates and signs the token when `allowed=True`.
3. The connectivity layer is configured separately.

The signing key is set in `harness.yaml`:

```yaml
# Future field — not yet implemented
connectivity:
  dispatch_token_secret: "secret://DISPATCH_TOKEN_SECRET"
  token_ttl_seconds: 10
```
